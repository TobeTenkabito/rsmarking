RSMarking Framework
===================

RSMarking is a remote-sensing annotation and raster analysis workspace.
The current product is a static browser client backed by FastAPI services,
PostgreSQL/PostGIS storage, GDAL/rasterio processing helpers, an AI gateway,
and an optional Celery worker layer for offline jobs.


Runtime Shape
-------------

Browser client
  - Leaflet and Cesium map views
  - Raster, vector, analysis, export, AI, and script modules
  - Served from client/index.html or from the data service static mount

FastAPI services
  - 8001 annotation_service: vector projects, layers, features, fields,
    shapefile import, and spatial vector operations
  - 8002 data_service: raster uploads, metadata, raster fields, algorithms,
    statistics, raster/vector conversion, GIS file export, and client static files
  - 8003 vtile_service: PostGIS vector tiles in MVT/PBF form
  - 8004 executor_service: Docker-isolated user Python script execution
  - 8005 tile_service: raster XYZ PNG tile rendering
  - 8006 ai_gateway: AI analyze/modify workflows and callable algorithm tools

Infrastructure
  - PostgreSQL/PostGIS stores raster metadata and vector data
  - storage/raw stores uploaded rasters and temporary script outputs
  - storage/cog stores cloud-optimized raster outputs
  - RabbitMQ and Redis support worker_cluster when offline jobs are used


Communication Map
-----------------

client/index.html
  |
  +-- raster API calls use the data service origin on 8002
  +-- VectorAPI calls annotation_service on 8001
  +-- VectorAPI builds vector tile URLs for vtile_service on 8003
  +-- MapEngine builds raster tile URLs for tile_service on 8005
  +-- AIAPI calls ai_gateway on 8006

data_service
  |
  +-- vector_bridge calls annotation_service on 8001 when a raster workflow
  |   needs vector layers, vector fields, or exported vector features
  +-- executor_bridge calls executor_service on 8004 for custom scripts
  +-- processor.py delegates numerical work to functions/implement

worker_cluster
  |
  +-- consumes Celery tasks from RabbitMQ when explicitly adopted
  +-- reports task status through Redis and optional task job persistence
  +-- is present and useful, but most current data_service routes still run
      processing synchronously

Client Workflows
----------------

Important client modules under client/packages/app/src/modules:
  - RasterModule and RasterStatisticsModule manage raster loading and inspection
  - AnalysisModule and CalculatorModule run indices and raster expressions
  - ExtractionModule and ChangeDetectionModule run mask and change workflows
  - AnnotationModule, ProjectModule, and AttributeTable manage vector editing
  - ClipModule and ConversionModule bridge raster/vector workflows
  - ExportModule exports map views and GIS workspace files
  - ScriptModule submits custom raster scripts to the data/executor path
  - AIModule drives AI analysis, AI modifications, and AI function calls


Backend Responsibilities
------------------------

annotation_service
  - Owns vector projects, layers, features, attribute fields, and spatial queries
  - Uses PostGIS-backed models and routers for project, layer, and spatial APIs

data_service
  - Owns raster metadata, raster attributes, upload/storage records, and raster APIs
  - Routers cover upload, raster records, fields, indices, extraction, clipping,
    rasterization, change detection, scripts, and export
  - processor.py wraps algorithms, metadata extraction, raster calculator logic,
    spectrum queries, conversion, and raster/vector processing helpers
  - export_router.py can package active raster and vector content into GIS files
    while retaining attribute information for desktop GIS workflows

tile services
  - tile_service renders raster PNG tiles from raster records
  - vtile_service renders vector MVT tiles from PostGIS

executor_service
  - Receives validated script payloads from data_service
  - Runs scripts in an isolated Docker runtime and returns output metadata

ai_gateway
  - Validates analyze/modify requests
  - Builds AI context for raster or vector targets
  - Exposes registered functions for direct algorithm invocation


Algorithm Layer
---------------

functions/implement contains the reusable processing work:
  - spectral_indices.py for NDVI, NDWI, NDBI, and MNDWI-style calculations
  - extraction/ for vegetation, water, building, and cloud extraction
  - manipulation.py for band extraction and band merge operations
  - clip_ops.py for raster/vector clipping
  - rasterize_ops.py for raster-to-vector and vector-to-raster conversion
  - change_ops.py for change detection operations
  - io_ops.py and spatial_ops.py for raster IO, COG/overview work, and bounds helpers


Current Notes
-------------

  - The browser application is currently a static client, not a package-managed
    dev-server application.
  - The active service ports are the six FastAPI ports listed above.
  - worker_cluster is optional until more data_service routes submit Celery jobs.
  - Tests now include unit coverage, bridge communication contracts, opt-in live
    port probes, benchmarks, and client Vitest units.
  - This file documents active project structure. Placeholder ideas that are not
    part of the current runtime are intentionally omitted.
